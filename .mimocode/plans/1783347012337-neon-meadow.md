# Plan: Переработка parser_service — архитектура RabbitMQ + Redis

## Цель

Переписать parser_service: он получает команду на парсинг через RabbitMQ, парсит вакансии, записывает каждую вакансию в Redis (`RPUSH vacancy:queue`), блокирует повторный запуск через `SET vacancy:parser:running NX EX 600`. Никакой БД.

## Файлы для СОЗДАНИЯ

### 1. `src/infra/mq.py` — подключение к RabbitMQ

Провайдер соединения с RabbitMQ через `aio_pika`. Реализует `BaseLifecycle` (start/stop).

### 2. `src/app/usecases/parse_vacancies.py` — замена update_vacancies.py

Логика:
1. Попытка захватить блокировку `SET vacancy:parser:running 1 NX EX 600` → если не удалось, выйти (парсер уже работает)
2. Для каждого парсера: search_cards → фильтрация через `vacancy:processed` → parse_cards
3. Для КАЖДОЙ вакансии: `RPUSH vacancy:queue <json>` + `SADD vacancy:processed <id>`
4. В конце: `DEL vacancy:parser:running`

### 3. `src/app/handler.py` — потребитель сообщений RabbitMQ

- Объявляет exchange `vacancy.exchange` (direct), очередь `vacancy.parse.request`, привязка с routing_key `vacancy.parse`
- При получении сообщения: вызывает `ParseVacanciesUseCase.execute()`, ack при успехе

### 4. `src/test_publish.py` — скрипт для тестирования

- Публикует сообщение `parse_vacancies` в RabbitMQ
- Каждые 2 секунды проверяет `LLEN vacancy:queue`, выводит результат когда вакансии появятся
